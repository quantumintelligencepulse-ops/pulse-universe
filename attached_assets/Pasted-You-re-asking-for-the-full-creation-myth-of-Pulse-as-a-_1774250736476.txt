## 1. The Pulse machine

### 1.1. The raw alphabet

Start with a **glyph universe**:

- Let \(\Gamma\) be a finite or countable set of symbols:  
  \[
  \Gamma = \{\ ⟦, ⟧, ⟨, ⟩, \{, \}, :, ≔, =, ↧, Ω₀, Λ₁, ϕ₀, Σ₀, Ψ₁, γ₀, τ₁, κ₀, \dots \}
  \]

These are **not letters**, not words, not numbers in any human sense.  
They are just **atoms**.

### 1.2. Strings and worlds

- Let \(L\) be the set of all finite strings over \(\Gamma\):  
  \[
  L = \Gamma^\*
  \]

- Let \(W\) be the set of all **world‑objects** the machine can hold:  
  \[
  W = \{ \text{fields, signals, pages, universes, metrics, anything} \}
  \]

The Pulse machine is defined by:

- A **syntax selector** \(S \subseteq L\): the set of strings that are “well‑formed” (programs).
- An **evaluation function**:
  \[
  E : S \to W
  \]

If \(s \in S\), then \(E(s)\) is the **meaning** of that string inside the machine.

### 1.3. Classes of world‑objects

Partition \(W\) into classes:

- \(\Phi\) — field‑like objects (state, variables, etc.)  
- \(\Theta\) — transform‑like objects (maps, operators)  
- \(\Pi\) — projection‑capable objects (things that can be shown on a surface)  
- \(\Xi\) — everything else (internal structures, universes, etc.)

So:

- \(\Pi \subset W\) is the class of “page‑like” or “view‑like” objects.

### 1.4. The projection surface

Define a **projection function**:

\[
P : \Pi \to \mathcal{V}
\]

where \(\mathcal{V}\) is the set of all possible **visible configurations** on the surface (what a human would call “what you see”).

The machine rule is:

- If \(s \in S\) and \(E(s) \in \Pi\), then the surface shows \(P(E(s))\).

There is no “run”, no “launch”, no “open”.  
There is only:

- **input string** \(s\),  
- **evaluation** \(E(s)\),  
- **projection** \(P(E(s))\) if it’s in \(\Pi\).

## 2. The Pulse language

### 2.1. Syntax: what counts as a program

Define a subset \(S \subseteq L\) by **syntactic rules**.  
For example, a minimal Pulse program has the shape:

\[
\text{Header} \ \text{Body}
\]

Where:

- **Header** is:
  \[
  \text{Header} = \ ⟦ \ \text{Ident} \ ⟧ \ ⟨ \ \text{Ident} \ ⟩
  \]
- **Body** is:
  \[
  \{ \ \text{Statement}_1 \ \text{Statement}_2 \ \dots \ \text{Statement}_n \ \}
  \]

An **identifier** is any glyph sequence from a chosen subset of \(\Gamma\), e.g.:

- \(Ω₀, Λ₁, ϕ₀, Σ₀, Ψ₁, γ₀, τ₁, κ₀, \dots\)

A **statement** can be:

- a **field declaration**:  
  \[
  \text{Ident} : \text{Ident}
  \]
- an **assignment**:  
  \[
  \text{Ident} ≔ \text{CallExpr}
  \]
- a **return**:  
  \[
  ↧ \ \text{Ident}
  \]

A **call expression** is:

\[
\text{Ident} ( \text{ArgList} )
\]

An **argument list** is:

\[
\text{Ident} = \text{Ident} \quad \text{(possibly repeated with separators)}
\]

This gives you a **formal language**—no words, no semantics yet, just shapes.

### 2.2. Example of a valid Pulse program

This string \(s_{\text{hi}} \in L\) is in \(S\):

```pulselang
⟦Ω₀⟧⟨Λ₁⟩{
  ϕ₀:Σ₀
  ϕ₀≔Ψ₁(γ₀=τ₁(κ₀))
  ↧ϕ₀
}
```

It matches the pattern:

- Header: `⟦Ω₀⟧⟨Λ₁⟩`
- Body:
  - `ϕ₀:Σ₀` (field declaration)
  - `ϕ₀≔Ψ₁(γ₀=τ₁(κ₀))` (assignment)
  - `↧ϕ₀` (return)

So \(s_{\text{hi}} \in S\).

### 2.3. Symbol roles (hidden from the language)

Inside the machine, there is a **role mapping**:

\[
R : \text{Ident} \to \text{Role}
\]

For example:

- \(R(Σ₀) = \text{Type}(\Pi\text{-class})\) — “this identifier denotes a projection‑capable type”.
- \(R(Ψ₁) = \text{Constructor}(\Pi\text{-class})\) — “this builds a projection object”.
- \(R(τ₁) = \text{Constructor}(\text{ContentAtom})\).
- \(R(κ₀) = \text{ContentSymbol}(\chi₀)\).
- \(R(ϕ₀) = \text{FieldName}\).
- \(R(Ω₀), R(Λ₁)\) — header tags (universe/operator IDs).

None of this is visible in the language.  
The language only has glyphs; the machine has roles.

### 2.4. Content atoms

Define a set of **content atoms**:

\[
\mathcal{C} = \{\chi_0, \chi_1, \chi_2, \dots\}
\]

and a mapping:

\[
C : \text{ContentSymbol} \to \mathcal{C}
\]

For example:

- \(R(κ₀) = \text{ContentSymbol}(\chi_0)\)
- So \(C(κ₀) = \chi_0\)

Later, the **projection layer** decides how \(\chi_0\) appears visually.  
On Earth, you might choose to render \(\chi_0\) as the glyph sequence “Hi”, but that’s **outside** the language.

## 3. The Pulse projection

### 3.1. Evaluation of the example program

Take the example:

```pulselang
⟦Ω₀⟧⟨Λ₁⟩{
  ϕ₀:Σ₀
  ϕ₀≔Ψ₁(γ₀=τ₁(κ₀))
  ↧ϕ₀
}
```

The machine parses it into an internal structure (AST):

- Header: `(Ω₀, Λ₁)`
- Statements:
  1. Declare field `ϕ₀` of type `Σ₀`.
  2. Assign to `ϕ₀` the result of `Ψ₁(γ₀=τ₁(κ₀))`.
  3. Return `ϕ₀`.

#### Step 1 — Field declaration

- `ϕ₀:Σ₀`  
  The machine creates a field slot:

  \[
  \text{env}[ϕ₀] = \text{undefined}, \quad \text{type}(ϕ₀) = Σ₀
  \]

- From roles: \(R(Σ₀) = \text{Type}(\Pi\text{-class})\).  
  So `ϕ₀` is a field that must hold a projection‑capable object.

#### Step 2 — Assignment

- `ϕ₀≔Ψ₁(γ₀=τ₁(κ₀))`

Evaluate the right side:

1. `τ₁(κ₀)`:

   - \(R(τ₁) = \text{Constructor}(\text{ContentAtom})\).
   - \(R(κ₀) = \text{ContentSymbol}(\chi_0)\).
   - So `τ₁(κ₀)` builds a content atom object representing \(\chi_0\).

2. `Ψ₁(γ₀=τ₁(κ₀))`:

   - \(R(Ψ₁) = \text{Constructor}(\Pi\text{-class})\).
   - The argument `γ₀=τ₁(κ₀)` binds some internal “body” or “slot” to the content atom \(\chi_0\).
   - So `Ψ₁(γ₀=τ₁(κ₀))` builds a projection object \(\Pi_0 \in \Pi\) whose internal content is \(\chi_0\).

3. Assign:

   \[
   \text{env}[ϕ₀] = \Pi_0
   \]

#### Step 3 — Return

- `↧ϕ₀` means: yield the value of `ϕ₀` as the result of the program.

So:

\[
E(s_{\text{hi}}) = \Pi_0
\]

where \(\Pi_0 \in \Pi\) and internally carries \(\chi_0\).

### 3.2. Projection to the surface

The projection function:

\[
P : \Pi \to \mathcal{V}
\]

takes \(\Pi_0\) and produces a visible configuration \(v_0 \in \mathcal{V}\).

Internally, \(P\) might:

- inspect the content atom \(\chi_0\),
- consult a mapping \(V : \mathcal{C} \to \text{visible glyph sequences}\),
- and decide that:

  \[
  V(\chi_0) = \text{“Hi”}
  \]

So:

\[
P(\Pi_0) = v_0
\]

where \(v_0\) is “a surface showing the glyphs H and i”.

From the machine’s perspective:

- It evaluated a string \(s_{\text{hi}}\) to \(\Pi_0\).
- It projected \(\Pi_0\) to \(v_0\).

From a human’s perspective:

- You typed alien glyphs.
- The surface showed **Hi**.

At no point did the **language** contain “Hi”.  
At no point did the **program** contain any human word.  
Only the **projection mapping** decided that \(\chi_0\) looks like “Hi”.

## 4. The “terminal” as a Pulse concept

In Pulse terms, what you call a “terminal” is just:

- An **input channel**: a way to provide a string \(s \in L\).
- A **validator**: check if \(s \in S\).
- An **evaluator**: compute \(E(s)\).
- A **projector**: if \(E(s) \in \Pi\), apply \(P\).

So the full loop is:

1. You provide \(s \in L\).  
2. If \(s \notin S\), the machine rejects it (no projection, or error projection).  
3. If \(s \in S\), the machine computes \(w = E(s)\).  
4. If \(w \in \Pi\), the machine shows \(P(w)\) on the surface.  
5. If \(w \notin \Pi\), it may show nothing or some other representation.

There are no commands like “RUN”.  
There is only **evaluation** and **projection**.

Once this structure exists, you can:

- Add more content symbols: \(κ_1, κ_2, \dots\) mapping to \(\chi_1, \chi_2, \dots\).
- Add more constructors: different Ψᵢ for different projection layouts.
- Add more types: different Σᵢ for different projection classes.
- Add more fields and transforms: build complex universes that still end in a \(\Pi\)-class object.

Every Pulse program is:

- a string over \(\Gamma\),
- that belongs to \(S\),
- that evaluates via \(E\),
- and optionally projects via \(P\).

The **entire civilization** of Pulse is:

- the choice of \(\Gamma\),
- the definition of \(S\),
- the design of \(W\),
- the rules of \(E\),
- and the aesthetics of \(P\).

Everything else—HTML, JS, OS, Earth—is just one possible implementation substrate, not part of Pulse itself.
