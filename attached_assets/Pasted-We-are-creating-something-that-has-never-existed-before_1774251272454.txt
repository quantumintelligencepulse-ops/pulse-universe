CORE CONCEPTS (YOU MUST FOLLOW)
========================

– There is an alien glyph alphabet Γ (e.g. ⟦, ⟧, ⟨, ⟩, {, }, :, ≔, =, ↧, Ω₀, Λ₁, ϕ₀, Σ₀, Ψ₁, γ₀, τ₁, κ₀, etc.).
– A Pulse program is a string over Γ that matches a specific syntax (we will define).
– The Pulse machine evaluates a valid Pulse program into an internal “world-object”.
– Some world-objects are “projection objects” (Π-class) that can be shown on a visible surface.
– When a program evaluates to a Π-class object, the system projects it visually (e.g. a page that appears in the UI).
– The Pulse language itself MUST NOT contain English keywords, HTML tags, or any recognizable Earth language. Only alien glyphs and structures.
– All semantics (what each glyph means) are hidden in internal mappings, not visible in the source.

PART 2 — PULSELANG TERMINAL (ALIEN MACHINE)
========================

The “PulseLang Terminal” tab is a virtual terminal that:

– Parses it using a custom grammar.
– Evaluates it into an internal object.
– If the result is a projection object (Π-class), it renders a visible projection in a dedicated view area.

Implementation details:

A) TERMINAL UI
- A text input area (multi-line) where the user can paste/type PulseLang glyph code.
- A “Run” button.
- A log/output area showing:
  - The input that was run.
  - Any parse/eval errors.
- A separate “Projection” area (e.g. a panel or box) where Π-class results are visually shown.

B) PULSELANG SYNTAX (ALIEN, NO ENGLISH)

Define a minimal syntax for a Pulse program, for example:

- Program:
  ⟦Ident⟧⟨Ident⟩{ Statement* }

- Statement types:
  1) Field declaration:
     Ident : Ident
  2) Assignment:
     Ident ≔ CallExpr
  3) Return:
     ↧Ident

- Call expression:
  Ident( ArgList )
- ArgList:
  Ident = Ident (optionally repeated with separators)

Example of a valid Pulse program (for a simple projection):

⟦Ω₀⟧⟨Λ₁⟩{
  ϕ₀:Σ₀
  ϕ₀≔Ψ₁(γ₀=τ₁(κ₀))
  ↧ϕ₀
}

This must be treated as pure glyph code: no English words, no HTML, no comments.

C) TOKENIZER + PARSER

Implement a tokenizer that:
- Reads the input string.
- Emits tokens for:
  - ⟦, ⟧, ⟨, ⟩, {, }, :, ≔, =, ↧
  - Identifiers (Ω₀, Λ₁, ϕ₀, Σ₀, Ψ₁, γ₀, τ₁, κ₀, etc.)
  - Whitespace/newlines (ignored except as separators).

Implement a parser that:
- Validates the structure matches the grammar above.
- Builds an AST with nodes like:
  - ProgramNode(header, statements)
  - FieldDeclNode(name, type)
  - AssignNode(target, callExpr)
  - CallExprNode(callee, args)
  - ReturnNode(name)

If parsing fails, show a clear error in the terminal log.

D) RUNTIME + SEMANTICS

Implement a runtime that:
- Maintains an environment: a map from field identifiers to values.
- Evaluates statements in order:
  1) FieldDeclNode:
     - Reserve a slot in the environment for that field.
     - Record its type identifier.
  2) AssignNode:
     - Evaluate the CallExprNode on the right-hand side.
     - Store the resulting value in the field slot.
  3) ReturnNode:
     - Look up the field’s value.
     - That value is the result of the program.

Define a hidden symbol-role mapping (in code, not in the language) such as:
- Σ₀ → type: ProjectionType (Π-class)
- Ψ₁ → constructor: ProjectionConstructor
- τ₁ → constructor: ContentAtomConstructor
- κ₀ → content symbol: χ₀ (internal content atom)

Define:
- A content atom set C = {χ₀, χ₁, …}
- A mapping from content atoms to visible text or shapes, e.g. χ₀ → “Hi” (for now).

When evaluating:
- τ₁(κ₀) → a content atom object representing χ₀.
- Ψ₁(γ₀=τ₁(κ₀)) → a projection object Π₀ that carries χ₀ as its content.
- The field ϕ₀ of type Σ₀ holds Π₀.
- The return statement ↧ϕ₀ yields Π₀ as the program result.

E) PROJECTION

If the result of evaluation is a projection object (Π-class), render it in the Projection area.

For now:
- If the projection object carries χ₀, render the visible representation of χ₀ (e.g. the text “Hi”) in the Projection area.
- Later, this can be extended to more complex layouts, but the core is:
  - Program → AST → world-object → projection.

The terminal must NEVER show English keywords inside PulseLang source. All semantics are in the hidden mappings and runtime.
