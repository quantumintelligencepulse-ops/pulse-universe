ENGINE IMPLEMENTATION (TYPESCRIPT)
========================================

Create a PulseLang engine under something like: client/src/lib/pulselang/

Files:

- tokenizer.ts
  - Input: raw PulseLang source (string of glyphs).
  - Output: token stream (tokens for ⟦, ⟧, ⟨, ⟩, {, }, :, ≔, ↧, ⊕, identifiers, etc.).

- parser.ts
  - Input: token stream.
  - Output: AST (ProgramNode, FieldDeclNode, AssignNode, CallExprNode, ReturnNode, etc.).
  - Enforce the grammar:
    - Program: Header + Body
    - Header: ⟦Ident⟧⟨Ident⟩
    - Body: { Statement* }
    - Statement: FieldDecl | Assign | Return
    - FieldDecl: Ident : Ident
    - Assign: Ident ≔ CallExpr
    - CallExpr: Ident( ArgList )
    - ArgList: Ident = Ident (possibly repeated)
    - Return: ↧Ident

- ast.ts
  - Define AST node classes/interfaces.

- symbolTable.ts
  - Define the hidden mapping from glyphs to roles:
    - Types (Σ₀–Σ₅)
    - Constructors (Ψ₀–Ψ₅)
    - Content constructors (τ₀–τ₉)
    - Content atoms (κ₀–κ₉ → χ₀–χ₉)
    - Special symbols (↧, ≔, ⊕, etc.).

- runtime.ts
  - Input: AST.
  - Output: world-object (with class info).
  - Maintain an environment (fields map).
  - Evaluate:
    - FieldDeclNode: allocate field with type.
    - AssignNode: evaluate CallExprNode, store in field.
    - CallExprNode: use symbolTable to determine which constructor to call (e.g. Ψ₁ builds a ΠPage object).
    - ReturnNode: return the field’s value as the program result.

- projector.ts
  - Input: world-object.
  - Output: a renderable representation for the UI.
  - For ΠPage (Σ₀):
    - Inspect its content atoms (χ₀, χ₁, etc.).
    - Map them to visible text or shapes.
    - Render them in the PROJECTION SURFACE panel.
